Input Fields

Input fields allow users to enter data. They have a standardized height and corner radius treatment, but are flexible in width. Input fields are based on the grid.

Input fields typically appear in forms and dialogs. Its design provides a clear affordance for interaction, making the fields discoverable in layouts, efficient to fill in, and accessible.
Providing the right type of the input field for requested data will help users enter information in the right format and avoid mistakes, making the process as easy and efficient as possible.

  • If any validation is needed, you can display this additionally.
  • To display a calendar in input fields see the date picker and its options.

Input fields have different states to guide users when interacting with them. The main advantage with this input field variant is that additional descriptions for conventions (e.g. telephone number) can be given.

There can be validations on input fields: A green tick appears when the content is validated. E.g. if the system has to validate the user’s input of the “@” within an email address.

An error message occurs if the user’s input is invalid or breaks input rules, such as special characters and passwords (which should be explained to the user to avoid errors in advance).

Date input fields

There are two different ways to enter a date: one with a masked date input field and one with date picker.

Masked date input fields

To enter dates that are far in the past, like a date of birth, please use the date input field split into three dedicated fields for day, month and year. Entering the date directly is the most efficient way if the date is further in the past or future.

Date input fields with date picker

For dates which are close to the present date (e.g. appointments) or for entering date range use the date picker variant.

Specifications

Autocompletion for input fields

Autocompletion speeds up the users interaction with the computer when it correctly predicts the word the user intends to enter – after 3 characters have been typed into the text field (mostly, this content is dynamic and comes from a database).
Whereas a filterable drop-down allows the users to filter a large list of choices – like a country selector – by typing in some characters (mostly, this content is static within the website).

Rules

  • For dates that are far off, for example date of birth, use the masked date input field instead.
  • Keep it short by eliminating unnecessary fields. Cutting fields means increasing conversion rate and reducing users' effort.
  • Group related labels and fields visually (e.g. if you ask two topics separate these groups of fields).
  • Display fields in a single column layout as multiple columns interrupt the vertical moving down the form. Exceptions are short and logical fields as city, state and zip code.
  • For related components next to each other use a custom Standalone Validation to display error messages instead of the default validation messages.
  • Use logical sequencing (e.g. credit card number, expiration date and its security code).
  • Use placeholder texts wisely and not per default. They are just recommended if the user has to enter specific formats (e.g. telephone number or birthday).
  • Distinguish optional and required fields. Put the instructions at the beginning of the form.
  • Avoid "reset" and "clear" buttons - it prevents the user from accidential deletion.
  • Give highly visible and specific error messages.
  • Match fields to input length:
    To achieve good usability adjust the size of form fields to match the length of the expected input (users start wondering if they had misunderstood the label if it is too short or too long). Form fields for variable input (e.g. name or email) typically range from 18 to 33 characters.

See rules and form example for how to use input fields.

Read-only and disabled

If you want to block the user from changing an input field add the readonly or disabled attribute to the sdx-input element.

The difference between the readonly and the disabled attribute is that in readonly the user can tab to the element.

Validation

Display validation information to the users by using the attributes valid and validation-message.

Pass the required attribute to the input field to denote them as required and make sure to explain the meaning of the asterisk in a legend below the form in the current customer language. It's recommended not to use any optional input fields and only ask the users to enter data that is really required.

Standalone validation

The sdx-validation-message component can be used to show a standalone validation message which can refer to multiple components. It should not be used in combination with validation-message attribute of sdx-input.

Automatic validation

Usually, SDX leaves the validation up to the consumers. But if there are enough heuristics, e.g. in <sdx-input-mask> SDX sets the valid and validationMessage properties itself. However, it only does that when both of them are unoccupied, meaning undefined. This way, the consumer is always in control of the validation.
