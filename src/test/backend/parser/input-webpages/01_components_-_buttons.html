Buttons

Buttons are graphical control elements that provide users a triggered event. They must contain a clear actionable text and may additionally contain icons. Buttons are more visually dominant than text links. They guide the user through the content as a visual highlight and act as "call to action".

For an example of how to use buttons in a form, please see layout variants.

Button group

The Button Group can be used to group buttons inside a container in order to create a set of related actions. The buttons will always be right aligned and will be placed under each other on mobile.
A Button Group can have an optional "Cancel" button to interrupt a certain operation. The cancel button has a specific style in order to be less prominent than the other buttons.

See also button group behaviours for additional layout options.

Button types

A screen can contain different button types. The hyerarchical structure can be controlled by using the different buttons.

  • Primary buttons are used for the main call to action and therefore use it sparingly and selectively.
  • Secondary buttons can be used in conjunction with primary buttons to indicate that the button is of lower priority.
  • Confirmation buttons are used for the final step in a process to indicate that an action is final and significant. For example, an 'Order' button at the end of a checkout process.
  • Icons can be used in buttons to help give their functionality more clarity at a glance.

Layout variants

The regular placement of the buttons on desktop, is bottom right.

Layout fullwidth

Within small components, such as cards, it's possible to use the fullwidth layout. This layout behaves the same on desktop and mobile screens.

Layout fill

Layout fill can be used inside small modals to place two buttons next to each other along the width. This variant is preferred to give equal importance to both actions. This layout should be used only with short labels that fit on mobile screens.

Layout responsive fill

This variant is based on "Layout fill" with the addition of a responsive behavior on mobile when the button labels require more text.

Button types

There are four button types: Primary, secondary, confirm and cancel.

Outlined buttons

For buttons on coloured background, set the attribute background="dark", so the basic button will just have an outline.

Transparent button

This button has no border at all and can be used for example for icon buttons or tooltip toggles. Use theme="transparent" for interactive single icons. Provide enough space for click and tap area.

This theme can be used with an icon and text or only icon.

Button group behaviours

Buttons can be placed inside an <sdx-button-group>. The Button Group then defines the layout and size of the buttons.

  • Standalone buttons can be placed without a button group.
  • Button groups with the responsive layout are always right aligned and reversed in order. This makes the first button in the DOM (primary) displayed as the first button on the right on desktops and on the top on mobiles.
  • To make the buttons use the full width of a group, use the layout fill.
  • Use the layout fullwidth to make every button use 100% of the width of the container. Buttons will be vertically aligned. Full width buttons should only be used in smaller containers, i.e. the sidebar or inside cards.
  • To keep the buttons from resizing, use the fixed layout.
  • A combination of "fill" and "responsive" is responsive-fill. On desktop the buttons have full width and on mobiles they are among each other.

Button as a link

To get a link in button look, just add the href attribute with value.

Semantic markup

Screen readers can differentiate between buttons and links and offer alternative control options for different HTML elements. You should always respect the true nature of an element and pass it on to the screen reader accordingly.

Buttons are generally used to trigger actions, such submitting forms, going forward in a process, open modals etc. For less important actions or to navigate to a new page use the link element.

  • Ensure to use the right markup for the correct purpose.
  • Define a clear actionable text.
  • If the URL has to navigate to new window or tab, provide a spoken warning into sr-hint to mitigate confusion for assistive technology.

Remember to put an appropriate text for the screen reader for the sr-hint attribute.
